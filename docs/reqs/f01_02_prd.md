## 1. Resumen Ejecutivo

Esta etapa verifica que la infraestructura técnica necesaria para el pipeline de datos está operativa y cumple con los contratos definidos en `CLAUDE.md §8`. No produce lógica de negocio ni transformaciones de datos; produce el **conector Python a Supabase** y la **documentación oficial del esquema de base de datos** (`docs/database/schema.sql`) como prerequisito técnico de toda la Fase 2.

El problema que resuelve: sin verificar la conectividad y el esquema antes de construir el pipeline, se descubren inconsistencias de infraestructura en mitad del desarrollo técnico, generando retrabajo costoso y riesgo de pérdida de datos. Esta etapa cierra esa brecha antes de escribir una sola línea de lógica de negocio.

**Entregables concretos de la etapa:**
- Ambiente virtual Python (`venv`) con todas las dependencias del stack declaradas en `requirements.txt`.
- Módulo `pipeline/src/supabase_client.py` — conector reutilizable como fundación de la Fase 2.
- Verificación documentada de que las 4 tablas `usr_*` del cliente existen en Supabase con el esquema correcto.
- Tablas operativas de Triple S creadas en Supabase: `tss_error_log`, `tss_pipeline_log`, `tss_quarantine`.
- `docs/database/schema.sql` poblado con el DDL completo verificado (tablas `usr_*` y `tss_*`).
- Suite de tests de integración que validan conectividad y esquema contra Supabase real.

## 7. Riesgos y Supuestos

| ID | Descripción | Probabilidad | Mitigación |
|---|---|---|---|
| `[RSK-01]` | Las tablas `usr_*` aún no existen en Supabase porque el cliente no las ha creado. | Media | Si las tablas no existen: (1) reportar las tablas faltantes al usuario, (2) proveer el DDL de `CLAUDE.md §8` listo para ejecutar en Supabase, (3) esperar confirmación antes de continuar. La etapa no puede cerrarse sin las 4 tablas verificadas. |
| `[RSK-02]` | Las credenciales del `.env` son incorrectas, el proyecto Supabase está inactivo o la URL no es accesible desde la red local. | Media | Verificar las variables del `.env` como primer paso (antes de cualquier conexión). Si la conexión falla, reportar el error exacto de `supabase-py` y pedir al usuario que verifique el panel de Supabase. No continuar con pasos de verificación de esquema hasta que la conectividad esté confirmada. |
| `[RSK-03]` | El esquema real de las tablas `usr_*` en Supabase difiere del DDL documentado en `CLAUDE.md §8` (el cliente modificó tablas sin informar). | Baja | Si se detecta discrepancia: (1) documentar la diferencia exacta (tabla, columna, tipo esperado vs real), (2) detener la etapa y activar `/change-control` antes de decidir si se actualiza el contrato o se solicita al cliente corregir su esquema. Nunca ignorar discrepancias silenciosamente. |
