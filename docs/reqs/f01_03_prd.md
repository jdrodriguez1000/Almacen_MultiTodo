## 1. Resumen Ejecutivo

La Etapa 1.2 confirmó que la infraestructura Supabase existe y es accesible: las 4 tablas del cliente (`usr_ventas`, `usr_inventario`, `usr_productos`, `usr_sedes`) están creadas con el esquema correcto y el pipeline tiene conectividad funcional.

Sin embargo, la existencia del esquema no garantiza la calidad de los datos que el cliente carga. Sin un contrato formal que defina exactamente qué se espera — en qué formato, con qué restricciones, en qué ventana horaria — el pipeline no tiene criterios objetivos para decidir si un dato es válido o no. El resultado sería un dashboard que muestra cifras incorrectas sin saberlo, o que falla silenciosamente sin informar al cliente la causa.

Esta etapa produce el **Contrato de Datos** entre Almacén MultiTodo y Triple S: un acuerdo técnico y operativo, vinculante para ambas partes, que especifica qué datos debe entregar el cliente, cuándo, con qué restricciones y qué ocurre cuando algo falla. Es el prerequisito técnico de la Fase 2 (ingeniería de datos).

## 5. Requerimientos de Datos

| ID | Fuente | Descripción | Formato Esperado |

| `[DAT-02]` | Supabase — `usr_inventario` | Stock físico por SKU/sede. Se sobreescribe en cada actualización diaria. | `sku` + `id_sede` (PK compuesta), `stock_fisico` (int >= 0), `costo_reposicion` (numeric > 0, nullable) |
| `[DAT-03]` | Supabase — `usr_productos` | Maestro de SKUs con jerarquía. Referenciado por FK desde `usr_ventas` e `usr_inventario`. | `sku` (text PK), `nombre` (text not null), `familia`, `categoria`, `subcategoria` (text not null) |
| `[DAT-04]` | Supabase — `usr_sedes` | Maestro de puntos de venta. 7 sedes activas en Colombia. | `id_sede` (serial PK), `pais` (text, default 'Colombia'), `ciudad` (text), `nombre_sede` (text) |
| `[DAT-05]` | Supabase — `tss_error_log` | Registro persistente de errores de validación. Escrito por el pipeline (service_role). | `error_code` (ERR_MTD_XXX), `error_message`, `source_table`, `source_record` (jsonb), `pipeline_run_id` |
| `[DAT-06]` | Supabase — `tss_quarantine` | Destino de registros rechazados. Permanecen hasta que el cliente corrija y reentregue. | `source_table`, `source_record` (jsonb), `error_code`, `error_detail`, `pipeline_run_id`, `resolved_at` (null hasta resolución) |

## 7. Riesgos y Supuestos

### 7.1 Riesgos

| ID | Descripción | Probabilidad | Impacto | Mitigación |
|---|---|---|---|---|
| `[RSK-01]` | El cliente no tiene un proceso definido para cargar datos antes de las 05:30 UTC | Alta | Alto | Incluir la ventana de entrega como compromiso explícito en el contrato. El dashboard muestra alerta visible cuando los datos no se actualizan. |
| `[RSK-02]` | El cliente carga datos del día en curso (T+0) antes del cierre de la jornada, generando registros parciales | Media | Alto | `ERR_MTD_004` rechaza automáticamente registros con `fecha_hora` del día en curso. Se retienen en cuarentena hasta el día siguiente. |
| `[RSK-03]` | El cliente tiene SKUs en `usr_ventas` que no existen en `usr_productos` por desactualización del maestro | Alta | Medio | `ERR_MTD_002` rechaza estas ventas. El protocolo de notificación informa al cliente qué SKUs faltan en el maestro. |
| `[RSK-04]` | El cliente modifica el esquema de una tabla `usr_*` sin notificar a Triple S | Baja | Muy Alto | Cualquier modificación de esquema requiere CC aprobado. El pipeline detecta discrepancias de esquema en la siguiente ejecución. |
| `[RSK-05]` | La notificación de violaciones no llega a la persona correcta del cliente, retrasando la corrección | Media | Medio | Definir en el contrato los canales y personas de contacto para recibir notificaciones de error. |
| `[RSK-06]` | El volumen de registros en cuarentena crece indefinidamente si el cliente no corrige | Baja | Medio | `tss_quarantine.resolved_at` rastreable. Proceso de limpieza periódica se define en Etapa 4.1. |

### 7.2 Supuestos

| ID | Supuesto |
|---|---|
| `[SUP-01]` | El cliente tiene capacidad técnica para cargar datos en las tablas `usr_*` de Supabase de forma diaria y automatizada. |
| `[SUP-02]` | Las 7 sedes listadas son la cobertura completa al momento de firmar el contrato. Nuevas sedes se agregan en `usr_sedes` antes de generar transacciones. |
| `[SUP-03]` | El cliente puede respetar la ventana operativa de 13:00–23:00 UTC para timestamps de ventas. |
| `[SUP-04]` | Los campos `costo` (ventas) y `costo_reposicion` (inventario) son opcionales. El cliente puede no tenerlos disponibles en una primera fase. |
| `[SUP-05]` | Todos los timestamps en `usr_ventas.fecha_hora` están en UTC. Si el sistema del cliente registra en hora local COT, el cliente es responsable de convertir antes de cargar. |
