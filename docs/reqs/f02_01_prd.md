## 1. Resumen Ejecutivo

La Fase 1 entregó los cimientos completos del proyecto: infraestructura Supabase verificada, conectividad funcional y un contrato de datos formal que define exactamente qué información entregará el cliente y bajo qué restricciones. Todo el andamiaje técnico está listo para construir.

Sin embargo, antes de invertir semanas en pipelines de validación, procesos ETL y cálculos de capa Gold, existe un riesgo crítico: el cliente podría aprobar el diseño visual del dashboard solo cuando ya esté construido, detectando en ese momento que el layout, las métricas visibles o la organización de la información no se ajustan a sus necesidades operativas reales. Corregir el diseño en ese punto implica reescribir componentes ya integrados con datos reales, con un costo de retrabajo significativo.

Esta etapa mitiga ese riesgo mediante una estrategia de **Mockup Primero**: construir un prototipo visual navegable en el navegador con datos ficticios, cuyo único propósito es que el cliente vea, navegue y apruebe la dirección de diseño del dashboard antes de que se construya la versión real. El mockup es un artefacto desechable — no llega a producción — pero su aprobación es el aval formal que desbloquea las Etapas 2.2, 2.3 y 2.4.

## 5. Requerimientos de Datos

> **Principio de Migración Cero:** Los JSON de mockup replican exactamente los esquemas de las tablas `usr_*` de Supabase (mismos nombres de columna, mismos tipos). Los campos derivados (clasificación ABC, días de cobertura, promedios) se colocan en archivos separados que representarán la capa Gold (`tss_gold_*`) en el sistema real. Este diseño garantiza que al conectar Supabase en Etapa 3.3, la capa de servicio es el único archivo que cambia — los tipos TypeScript y los componentes no se tocan.

| ID | Fuente | Descripción | Formato Esperado — refleja exactamente el esquema `usr_*` |
|---|---|---|---|
| `[DAT-01]` | JSON local — `mock_ventas.json` | Datos de ventas representativos: 7 sedes, ~100 SKUs, últimos 7 días ficticios + mismo período mes anterior. `fecha_hora` en UTC (formato ISO 8601), dentro de ventana operativa `13:00–23:00 UTC`. Mínimo 500 registros. | Array de objetos con columnas exactas de `usr_ventas`: `{ id_venta: number, fecha_hora: string (ISO UTC), sku: string, id_sede: number, cantidad: number, precio: number, costo: number \| null }` |
| `[DAT-02]` | JSON local — `mock_inventario.json` | Stock por SKU/sede: algunos en stock crítico (activa `ALT_NEG_001`), algunos sin stock (activa `ALT_NEG_004`), mayoría en rango normal. Cubre las 7 sedes y ~100 SKUs. | Array de objetos con columnas exactas de `usr_inventario`: `{ sku: string, id_sede: number, stock_fisico: number, costo_reposicion: number \| null }` |
| `[DAT-03]` | JSON local — `mock_productos.json` | Catálogo de ~100 SKUs con nombres representativos de abarrotes/consumo masivo colombiano, con jerarquía `familia → categoría → subcategoría`. | Array de objetos con columnas exactas de `usr_productos`: `{ sku: string, nombre: string, familia: string, categoria: string, subcategoria: string }` |
| `[DAT-04]` | JSON local — `mock_sedes.json` | Las 7 sedes de Almacén MultiTodo con sus ciudades y nombres. Datos de dominio público — no son datos operacionales. | Array de objetos con columnas exactas de `usr_sedes`: `{ id_sede: number, pais: string, ciudad: string, nombre_sede: string }`. Exactamente 7 registros. |
| `[DAT-05]` | JSON local — `mock_gold.json` | Métricas derivadas que en el sistema real provienen de la capa Gold (`tss_gold_*`): clasificación ABC por SKU, promedio de venta de 7 días por SKU/sede, días de cobertura estimados. Estos campos NO están en las tablas `usr_*` — están aquí para que el mockup pueda mostrarlos, y serán reemplazados por queries a Gold en Etapa 3.3. | Objeto con secciones: `{ clasificacion_abc: [{ sku, clase_abc }], metricas_inventario: [{ sku, id_sede, promedio_venta_7d, dias_cobertura }] }` |
| `[DAT-06]` | JSON local — `mock_alertas.json` | Las 12 alertas del sistema pre-calculadas con ejemplos representativos. En el sistema real se calcularán por el motor de alertas de Etapa 3.2 sobre la capa Gold. | Array de objetos: `{ codigo: string, tipo: 'negativa' \| 'positiva', severidad: 'critico' \| 'alto' \| 'medio', sku: string \| null, id_sede: number \| null, mensaje: string, fecha_calculo: string (ISO) }`. Exactamente 12 registros. |

## 7. Riesgos y Supuestos

### 7.1 Riesgos

| ID | Descripción | Probabilidad | Impacto | Mitigación |
|---|---|---|---|---|
| `[RSK-01]` | El cliente solicita cambios profundos al diseño después de ver el mockup, requiriendo una segunda iteración | Media | Medio | El mockup es barato de modificar (sin backend). Una segunda iteración de mockup es preferible a retrabajo en Etapa 3.3. Documentar cada cambio solicitado para alimentar la SPEC de 3.3. |
| `[RSK-02]` | El cliente confunde el mockup con el dashboard funcional y espera datos reales inmediatamente | Media | Alto | Comunicar explícitamente antes y durante la sesión de validación que el mockup es un prototipo con datos ficticios. Incluir un banner visible en el mockup: `"PROTOTIPO — Datos ficticios para validación de diseño"`. |
| `[RSK-03]` | Los datos ficticios no son suficientemente representativos y el cliente no puede dar feedback significativo | Media | Medio | Los JSON deben construirse con rangos y distribuciones que reflejen el negocio real (7 sedes, ~100 SKUs, proporciones ABC 20/30/50, alertas de todos los tipos). Revisar con el equipo que los datos "suenan reales". |
| `[RSK-04]` | El mockup tarda más de lo esperado en construirse, retrasando el inicio de Etapa 2.2 | Baja | Medio | El mockup es un artefacto deliberadamente simple: sin backend, sin lógica compleja. El riesgo se mitiga limitando el alcance a lo estrictamente necesario para la validación de diseño. |
| `[RSK-05]` | El cliente aprueba el mockup pero no asiste a la sesión de validación o delega a alguien sin autoridad para aprobar | Media | Alto | Coordinar con anticipación que la sesión de validación incluya a al menos un gerente con poder de decisión. Documentar el responsable de aprobación en el acta. |

### 7.2 Supuestos

| ID | Supuesto |
|---|---|
| `[SUP-01]` | El stack web elegido (Next.js + TypeScript + Tailwind CSS) está validado desde CLAUDE.md §3 y no requiere aprobación adicional para el mockup. |
| `[SUP-02]` | Los datos ficticios son construidos por Triple S sin información real del cliente — el cliente no entrega datos para el mockup. |
| `[SUP-03]` | La sesión de validación del mockup con el cliente se realiza en ambiente local (localhost) o mediante screen sharing — no se requiere deploy para la validación. |
